# <a name="azure-lab-services---administrator-guide"></a>Службы лабораторий Azure: руководство для администратора
ИТ-администраторы, управляющие облачными ресурсами университета, обычно отвечают за настройку учетной записи лаборатории для своего учебного заведения. После настройки учетной записи лаборатории Администраторы или преподаватели создают лаборатории, содержащиеся в учетной записи. В этой статье представлен общий обзор ресурсов Azure, которые участвуют в работе, и рекомендации по их созданию.

![Схема высокоуровневого представления ресурсов Azure в учетной записи лаборатории.](./media/administrator-guide/high-level-view.png)

- Лаборатории размещаются в подписке Azure, которая принадлежит службам лаборатории Azure.
- Учетные записи лаборатории, Общая Коллекция образов и версии изображений размещаются в вашей подписке.
- Вы можете создать свою учетную запись лаборатории и общую коллекцию образов в одной группе ресурсов. На приведенной ниже схеме они находятся в разных группах ресурсов.

Дополнительные сведения об архитектуре см. в статье [основы архитектуры лабораторий](./classroom-labs-fundamentals.md).

## <a name="subscription"></a>Подписка
В вашем университете может быть одна или несколько подписок Azure. Подписки используются для управления выставлением счетов и безопасностью для всех используемых в нем ресурсов и служб Azure, включая учетные записи лаборатории.

Связь между учетной записью лаборатории и ее подпиской важна, потому что:

- выставление счетов происходит через подписку, содержащую учетную запись лаборатории;
- Вы можете предоставить пользователям доступ к службам лаборатории Azure для клиента Azure Active Directory подписки (Azure AD). Вы можете добавить пользователя в качестве владельца или участника учетной записи лаборатории либо в качестве создателя лаборатории или владельца лаборатории.

Лабораторные работы и их виртуальные машины управляются и размещаются в рамках подписки, которая принадлежит службам лаборатории Azure.

## <a name="resource-group"></a>Группа ресурсов
Подписка содержит одну или несколько групп ресурсов. Группы ресурсов можно использовать, чтобы создавать логические группировки ресурсов Azure, используемых одновременно в одном решении.  

При создании учетной записи лаборатории вам необходимо настроить группу ресурсов, содержащую учетную запись лаборатории. 

Группа ресурсов также необходима при создании [коллекции общих образов](#shared-image-gallery). Учетную запись лаборатории и коллекцию общих образов можно разместить в одной группе ресурсов или в двух отдельных группах ресурсов. Этот второй подход можно использовать, если планируется совместно использовать коллекцию изображений в различных решениях.

При создании учетной записи лаборатории вы можете автоматически создать и одновременно вложить общую коллекцию образов.  Такой параметр позволит создать учетную запись лаборатории и общую коллекцию образов в отдельных группах ресурсов. Это поведение будет отображаться при выполнении действий, описанных в разделе [Настройка общей коллекции образов на момент создания учетной записи лаборатории](how-to-attach-detach-shared-image-gallery.md#configure-at-the-time-of-lab-account-creation) . Эта конфигурация используется в образе в начале этой статьи. 

Мы рекомендуем всегда тратить время на планирование структуры групп ресурсов, так как после создания учетной записи лаборатории или общей коллекции *образов невозможно изменить* группу ресурсов. Если необходимо изменить группу ресурсов для этих ресурсов, необходимо удалить и повторно создать учетную запись лаборатории или общую галерею образов.

## <a name="lab-account"></a>Учетная запись лаборатории

Учетная запись лаборатории выступает в качестве контейнера для одной или нескольких лабораторий. Когда вы начинаете работать со службами лаборатории Azure, чаще всего используется одна учетная запись лаборатории. По мере масштабирования использования в лабораторных условиях можно позднее создать дополнительные учетные записи лаборатории.

В следующем списке перечислены сценарии, в которых может оказаться полезным более одной учетной записи лаборатории:

- **Управление различными требованиями к политикам в лабораториях**

    При настройке учетной записи лаборатории задаются политики, которые применяются ко *всем* лабораториям в учетной записи лаборатории, например:
    - Виртуальная сеть Azure с общими ресурсами, к которым может получить доступ лаборатория. Например, у вас может быть ряд лабораторных занятий, которым необходим доступ к общему набору данных в виртуальной сети.
    - Образы виртуальных машин, которые лаборатории могут использовать для создания виртуальных машин. Например, у вас может быть ряд лабораторных занятий, которым необходим доступ к [виртуальной машине для обработки и анализа данных для Linux](https://azuremarketplace.microsoft.com/marketplace/apps/microsoft-dsvm.ubuntu-1804) Azure Marketplace.

    Если каждая из ваших лабораторных занятий имеет уникальные требования к политике, может оказаться полезным создание отдельных учетных записей лаборатории для управления каждой лабораторией отдельно.

- **Назначение отдельного бюджета для каждой учетной записи лаборатории**
  
    Вместо того чтобы сообщать все затраты на лабораторию через одну лабораторную учетную запись, может потребоваться более четко распределенный бюджет. Например, можно создать отдельные учетные записи лаборатории для математического отдела университета, отделов компьютерных технологий и т. д., чтобы распределить бюджет между отделами.  Вы можете просмотреть затраты для каждой отдельной учетной записи лаборатории с помощью службы " [Управление затратами Azure](../cost-management-billing/cost-management-billing-overview.md)".

- **Изоляция пилотных лабораторных занятий от активных и рабочих лабораторий**
  
    В некоторых случаях требуется пилотное тестирование изменений политики для учетной записи лаборатории без потенциального влияния на активные или рабочие лаборатории. В сценарии такого типа создание отдельной учетной записи лаборатории для пилотного проекта позволяет отделить изменения. 

## <a name="lab"></a>Лаборатория

Лаборатория содержит виртуальные машины, каждый из которых назначен одному студенту.  Как правило, вам будет предоставлена:

- Наличие одной лаборатории для каждого класса.
- Создайте новый набор лабораторных занятий для каждого полугодия, квартала или другой используемой вами системы учебного заведения. Для классов, которые должны использовать одно и то же изображение, следует использовать [коллекцию общих образов](#shared-image-gallery). Таким образом можно повторно использовать образы в лабораторных занятиях и учебных периодах.

При определении способа структурирования лабораторий необходимо учитывать следующие моменты.

- **Все виртуальные машины в лаборатории развертываются с тем же образом, который опубликован**

    В результате, если у вас есть класс, который требует одновременной публикации различных образов лаборатории, для каждого образа необходимо создать отдельную лабораторию.
  
- **Квота использования задается на уровне лаборатории и применяется ко всем пользователям в лаборатории.**

    Чтобы задать различные квоты для пользователей, необходимо создать отдельные лаборатории. Однако после задания квоты можно добавить дополнительные часы для конкретных пользователей.
  
- **Расписание запуска или завершения работы задается на уровне лаборатории и применяется ко всем виртуальным машинам в лаборатории**.

    Как и в случае с параметрами квоты, если необходимо задать различные расписания для пользователей, необходимо создать отдельную лабораторию для каждого расписания.

По умолчанию каждая лаборатория имеет собственную виртуальную сеть.  Если установлен пиринг виртуальных сетей, каждая лаборатория будет иметь собственную подсеть с указанной виртуальной сетью.

## <a name="shared-image-gallery"></a>Общая коллекция образов

Общая коллекция образов подключается к учетной записи лаборатории и служит центральным репозиторием для хранения образов. Изображение сохраняется в коллекции, когда преподаватель выбирает экспорт из виртуальной машины шаблона лаборатории. Каждый раз, когда преподаватель вносит изменения в виртуальную машину шаблона и экспортирует ее, новые версии образа сохраняются, а предыдущие версии сохраняются.

Преподаватели могут публиковать версию образа из коллекции общих образов при создании новой лабораторной работы. Хотя в галерее хранится несколько версий образа, преподаватели могут выбирать только последнюю версию во время создания лаборатории.

Служба "Коллекция общих образов" — это необязательный ресурс, который вам может не понадобиться сразу, если вы начинаете с нескольких лабораторий. Однако коллекция общих образов предоставляет множество преимуществ, которые полезны при масштабировании до дополнительных лабораторий.

- **Вы можете сохранять версии шаблона образа виртуальной машины и управлять ими**

    Полезно создать пользовательский образ или внести изменения (программное обеспечение, конфигурацию и т. д.) в образ из общедоступной коллекции Azure Marketplace.  Например, для преподавателей требуется установить другое программное обеспечение или инструментарий. Вместо того, чтобы студенты могли вручную устанавливать эти компоненты самостоятельно, разные версии шаблона образа виртуальной машины можно экспортировать в общую коллекцию образов. Затем эти версии образа можно использовать при создании новых лабораторий.

- **Вы можете совместно использовать и повторно использовать образы виртуальных машин в лабораторных занятиях.**

    Вы можете сохранить и повторно использовать образ, чтобы не настраивать его с нуля при каждом создании лабораторной работы. Например, если несколько классов должны использовать одно и то же изображение, его можно создать один раз и экспортировать в общую галерею образов, чтобы обеспечить общий доступ к ним в лабораторных занятиях.

- **Доступность образа обеспечивается за счет автоматической репликации**

    При сохранении образа из лабораторной среды в общую коллекцию образов она автоматически реплицируется в другие [регионы в пределах той же географической области](https://azure.microsoft.com/global-infrastructure/regions/). В случае сбоя в регионе Публикация образа в лаборатории не затрагивается, так как можно использовать реплику образа из другого региона.  Публикация виртуальных машин из нескольких реплик может также повышать производительность.

Для логической группировки общих образов можно выполнить одно из следующих действий.

- Создание нескольких общих коллекций образов. Каждая учетная запись лаборатории может подключаться только к одной коллекции общих образов, поэтому для этого параметра также необходимо создать несколько учетных записей лаборатории.
- Используйте одну коллекцию общих образов, совместно используемую несколькими учетными записями лаборатории. В этом случае каждая учетная запись лаборатории может включать только те образы, которые применимы к лабораториям в этой учетной записи.

## <a name="regionslocations"></a>Регионы или расположение

При настройке ресурсов служб лаборатории Azure необходимо предоставить регион или расположение центра обработки данных, в котором будут размещаться ресурсы. В следующих разделах описывается, как область или расположение могут повлиять на каждый ресурс, участвующий в настройке лаборатории.

### <a name="resource-group"></a>Группа ресурсов

Регион определяет центр обработки данных, где хранятся сведения о группе ресурсов. Ресурсы Azure, содержащиеся в группе ресурсов, могут находиться в другом регионе, отличном от родительского.

### <a name="lab-account"></a>Учетная запись лаборатории

Расположение учетной записи лаборатории указывает область, в которой находится ресурс.  

### <a name="lab"></a>Лаборатория

Расположение лаборатории отличается в зависимости от следующих факторов.

  - **Учетная запись лаборатории соединена с виртуальной сетью.**
  
    Вы можете выполнить [одноранговую учетную запись лаборатории с виртуальной сетью](./how-to-connect-peer-virtual-network.md) , если они находятся в одном регионе.  Если учетная запись лаборатории соединена с виртуальной сетью, лабораторные занятия автоматически создаются в том же регионе, что и учетная запись лаборатории и виртуальная сеть.

    > [!NOTE]
    > Если учетная запись лаборатории соединена с виртуальной сетью, то параметр **Разрешить создателю лаборатории выбрать расположение лаборатории** будет отключен. Дополнительные сведения см. в статье [Предоставление создателю лаборатории разрешения на выбор расположения лабораторий](./allow-lab-creator-pick-lab-location.md).
    
  - **Ни одна виртуальная сеть не является пиринга *, а* создатели лабораторий не могут выбрать расположение лаборатории.**
  
    Если *ни одна* виртуальная сеть не связана с учетной записью лаборатории и [создателям лаборатории *не разрешено* выбирать расположение лаборатории](./allow-lab-creator-pick-lab-location.md), лабораторные занятия автоматически создаются в регионе с доступной емкостью виртуальных машин.  В частности, поиск для служб лабораторий Azure происходит в [регионах, которые находятся в той же географии, что и учетная запись лаборатории](https://azure.microsoft.com/global-infrastructure/regions).

  - **Ни одна виртуальная сеть не является одноранговыми *, и* создателям лабораторий запрещено выбирать расположение лаборатории.**
       
    Если *ни одна* из виртуальных сетей не допускает [Выбор расположения *are allowed* лаборатории](./allow-lab-creator-pick-lab-location.md), то расположения, которые могут выбираться создателем лаборатории, зависят от доступной емкости.

> [!NOTE]
> Чтобы обеспечить достаточную емкость виртуальной машины в регионе, важно сначала запросить емкость через учетную запись лаборатории при создании лаборатории.

Общее правило заключается в том, чтобы задать в качестве региона ресурса ближайший к пользователям. Для лабораторных занятий это означает создание лабораторной работы, ближайшей к вашим учащимся. Для интерактивных курсов, учащихся которых находится по всему миру, используйте свои решения, чтобы создать лабораторию, которая централизованно размещена. Вы также можете разделить класс на несколько лабораторий в соответствии с регионами учащихся.

## <a name="vm-sizing"></a>Изменение размера виртуальной машины

## <a name="manage-identity"></a>Управление удостоверениями

Используя [Управление доступом на основе ролей (RBAC) Azure](../role-based-access-control/overview.md) для доступа к учетным записям лаборатории и лабораториям, можно назначить следующие роли:

- **Владелец** учетной записи лаборатории

    Администратор, создающий учетную запись лаборатории, автоматически назначает роль владельца учетной записи лаборатории. Роль владельца может:
     - Измените параметры учетной записи лаборатории.
     - Предоставьте другим администраторам доступ к учетной записи лаборатории в качестве владельца или участника.
     - Предоставьте преподавателям доступ к лабораториям в качестве создателя, владельца или участника.
     - Создание лабораторий и управление ими в учетной записи лаборатории.

- **Участник** учетной записи лаборатории

    Администратор, которому назначена роль участника, может:
    - Измените параметры учетной записи лаборатории.
    - Создание лабораторий и управление ими в учетной записи лаборатории.

    Однако участник *не может* предоставить другим пользователям доступ к учетным записям лаборатории или лабораториям.

- **Создатель лаборатории**

    Для создания лабораторий в рамках учетной записи лаборатории преподаватель должен быть членом роли "создатель лаборатории".  Преподаватель, создающий лабораторию, автоматически добавляется в качестве владельца лаборатории. Дополнительные сведения см. в разделе [Добавление пользователя в роль создателя лаборатории](./tutorial-setup-lab-account.md#add-a-user-to-the-lab-creator-role). 

- **Владелец** или **участник** лаборатории
  
    Преподаватель в роли владельца или участника лаборатории может просматривать и изменять параметры лаборатории. Пользователь также должен быть членом роли читателя учетной записи лаборатории.

    Основное различие между ролями владельца и участника лаборатории заключается в том, что только владелец может предоставить другим пользователям доступ для управления лабораторией. Участник *не может* предоставить другим пользователям доступ для управления лабораторией.

- **Общая коллекция образов**.

    При присоединении коллекции общих образов к учетной записи лаборатории владельцы учетных записей лаборатории, участники и создатели лабораторий, владельцы лабораторий и участники лабораторий автоматически получают доступ к просмотру и сохранению изображений в коллекции.

При назначении ролей полезно выполнить следующие советы:

   - Обычно только администраторы должны быть членами роли владельца или участника учетной записи лаборатории. Учетная запись лаборатории может иметь более одного владельца или участника.
   - Чтобы дать преподавателям возможность создавать новые лабораторные занятия и управлять лабораторными занятиями, их необходимо назначить только роли создателя лаборатории.
   - Чтобы дать преподавателям возможность управлять конкретными лабораториями, но *не* возможностью создавать новые лаборатории, назначьте им роль владельца или участника для каждой лабораторной работы, которой они будут управлять. Например, вы можете разрешить профессор и помощнику по обведению для совместной работы над лабораторией. Дополнительные сведения см. в статье [Добавление владельцев в лабораторию](./how-to-add-user-lab-owner.md).

## <a name="pricing"></a>Цены

### <a name="azure-lab-services"></a>Службы лабораторий Azure

Дополнительные сведения о ценах см. на странице [цен на службы лаборатории Azure](https://azure.microsoft.com/pricing/details/lab-services/).


### <a name="shared-image-gallery"></a>Общая коллекция образов

Также необходимо учитывать цены на службу коллекции общих образов, если планируется использовать общие галереи изображений для хранения версий образов и управления ими. 

Создание общей коллекции образов и ее вложение к учетной записи лаборатории осуществляется бесплатно. Пока вы не сохраните версию образа в галерее, затраты не взимается. Цены на использование общей коллекции образов обычно довольно незначительны, но важно понимать, как она рассчитывается, поскольку она не включена в цены на службы лабораторий Azure.  

#### <a name="storage-charges"></a>Расходы на хранение

Для хранения версий образов общая Коллекция образов использует стандартные диски, управляемые жесткими дисками (HDD). Размер используемого диска, управляемого дисками, зависит от размера хранимой версии образа. Дополнительные сведения о ценах см. на странице [цены на управляемые диски](https://azure.microsoft.com/pricing/details/managed-disks/).

#### <a name="replication-and-network-egress-charges"></a>Расходы за репликацию и исходящий трафик сети

При сохранении версии образа с помощью виртуальной машины лабораторных шаблонов службы Azure Lab Services сначала сохраняют его в исходном регионе, а затем автоматически реплицируют версию исходного образа в один или несколько целевых регионов. 

Важно отметить, что службы лаборатории Azure автоматически реплицируют версию исходного образа во все [целевые регионы в пределах географического](https://azure.microsoft.com/global-infrastructure/regions/) расположения лаборатории. Например, если лаборатория находится в географическом регионе США, то версия образа реплицируется в каждый из восьми регионов, существующих в США.

Плата за исходящий трафик сети взимается при репликации версии образа из исходного региона в дополнительные целевые регионы. Сумма счета зависит от размера версии образа, когда данные образа изначально передаются в качестве исходящего трафика из исходного региона.  Сведения о ценах см. в разделе [сведения о ценах на пропускную способность](https://azure.microsoft.com/pricing/details/bandwidth/).

Оплата за исходящий трафик может быть отменена для клиентов [учебных решений](https://www.microsoft.com/licensing/licensing-programs/licensing-for-industries?rtc=1&activetab=licensing-for-industries-pivot:primaryr3) . Чтобы узнать больше, обратитесь к своему менеджеру по учетным записям. 

Дополнительные сведения см. в разделе «какие программы передачи данных существуют для учебных заведений и как они определены?». в разделе часто задаваемых вопросов на странице [программы для образовательных учреждений](https://azure.microsoft.com/pricing/details/bandwidth/) .

#### <a name="pricing-example"></a>Пример цен

Рассмотрим пример стоимости сохранения шаблона образа виртуальной машины в общей коллекции образов. Рассмотрим следующие сценарии:

- доступен один пользовательский образ виртуальной машины;
- вы сохраняете две версии образа;
- Ваша лаборатория находится в США, в которой всего восемь регионов.
- размер каждой версии образа составляет 32 ГБ. В результате стоимость управляемого диска HDD, составляет 1,54 долл. США в месяц.

Итоговые затраты в месяц оцениваются следующим образом:

* *Число образов &times; номер версии число версий &times; реплик &times; управляемого диска Цена — Общая стоимость в месяц*

В этом примере стоимость рассчитывается как:

* 1 пользовательский образ (32 ГБ) &times; 2 версии &times; 8 регионы США &times; $1,54 = $24,64 в месяц

> [!NOTE]
> Приведенные выше вычисления предназначены только для примера. Она охватывает затраты на хранение, связанные с использованием общей коллекции образов, и *не* включает затраты на исходящий трафик. Сведения о фактических ценах на хранилище см. в разделе [цены на управляемые диски](https://azure.microsoft.com/en-us/pricing/details/managed-disks/).

#### <a name="cost-management"></a>управления затратами;

Администраторам учетных записей лаборатории важно управлять затратами с помощью регулярных операций удаления ненужных версий образов из коллекции. 

Не удаляйте репликацию в определенные регионы, чтобы снизить затраты, хотя этот параметр существует в коллекции общих образов. Изменения репликации могут оказать негативное влияние на способность служб лаборатории Azure публиковать виртуальные машины из образов, сохраненных в общей коллекции образов.
