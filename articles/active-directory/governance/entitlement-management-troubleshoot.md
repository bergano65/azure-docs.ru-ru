# <a name="troubleshoot-azure-ad-entitlement-management"></a>Устранение неполадок управления назначениями Azure AD

В этой статье описываются некоторые элементы, которые необходимо проверить, чтобы помочь в устранении неполадок управления обслуживанием Azure Active Directory (Azure AD).

## <a name="administration"></a>Администрирование

* Если при настройке управления назначением вы получаете сообщение об отказе в доступе и являетесь глобальным администратором, убедитесь, что в каталоге есть [лицензия Azure AD Premium P2 (или EMS)](entitlement-management-overview.md#license-requirements).

* При получении сообщения об отказе в доступе при создании или просмотре пакетов доступа, если вы являетесь членом группы создателей каталога, необходимо [создать каталог](entitlement-management-catalog-create.md) перед созданием первого пакета Access.

## <a name="resources"></a>Ресурсы

* Роли для приложений определяются в самом приложении, и управление ими осуществляется в Azure AD. Если приложение не имеет ролей ресурсов, управление назначением назначает пользователям роль **доступа по умолчанию** .

    Обратите внимание, что в портал Azure также могут отображаться субъекты-службы для служб, которые не могут быть выбраны в качестве приложений.  В частности, **Exchange Online** и **SharePoint Online** являются службами, а не приложениями, имеющими роли ресурсов в каталоге, поэтому они не могут быть включены в пакет Access.  Вместо этого используйте групповое лицензирование и установите подходящую лицензию для пользователей, которым нужен доступ к этим службам.

* Чтобы группа была ресурсом в пакете для доступа, она должна быть доступна для изменения в Azure AD.  Группы из локального каталога Active Directory нельзя назначить в качестве ресурсов, так как их атрибуты владельца и членов нельзя изменить в Azure AD.   Группы, созданные в Exchange Online в качестве списков рассылки, нельзя изменить в Azure AD. 

* Библиотеки документов SharePoint Online и отдельные документы нельзя добавить в качестве ресурсов.  Вместо этого создайте [группу безопасности Azure AD](../fundamentals/active-directory-groups-create-azure-portal.md), включите эту группу и роль сайта в пакет Access, а в SharePoint Online используйте эту группу для управления доступом к библиотеке документов или документу.

* Если есть пользователи, которые уже назначены ресурсу, которым вы хотите управлять с помощью пакета для доступа, убедитесь, что пользователи назначены пакету с использованием соответствующей политики. Например, может потребоваться включить группу в пакет для доступа, в котором уже есть пользователи в группе. Если пользователям в группе требуется постоянный доступ, в их отношении должна действовать соответствующая политика для пакетов для доступа, чтобы они не утратили доступ к группе. Чтобы назначить пакет для доступа, можно уведомить пользователей о необходимости запросить пакет, содержащий этот ресурс, или напрямую назначить их пакету. Дополнительные сведения см. в разделе [изменение параметров запросов и утверждений для пакета Access](entitlement-management-access-package-request-policy.md).

* При удалении члена команды они также удаляются из группы Microsoft 365. С удалением из чата команды может возникнуть задержка. Дополнительные сведения см. в разделе [членство в группе](/microsoftteams/office-365-groups#group-membership).

* Убедитесь, что каталог не настроен для работы в нескольких регионах. Сейчас для управления правами не поддерживаются несколько регионов для SharePoint Online. Чтобы можно было управлять правами для сайтов SharePoint Online, эти сайты должны принадлежать к географическому расположению по умолчанию. Дополнительные сведения см. [в статье поддержка нескольких географических возможностей в OneDrive и SharePoint Online](/Microsoft 365/Enterprise/multi-geo-capabilities-in-onedrive-and-sharepoint-online-in-office-365).

## <a name="access-packages"></a>Пакеты для доступа

* Если вы попытаетесь удалить пакет или политику доступа и видите сообщение об ошибке, в котором говорится о наличии активных назначений, то если вы не видите пользователей с назначениями, проверьте, не назначены ли им недавно удаленные пользователи. В течение 30-дневного периода после удаления пользователя учетная запись пользователя может быть восстановлена.   

## <a name="external-users"></a>Внешние пользователи

* Когда внешний пользователь хочет запросить доступ к пакету Access, убедитесь, что он использует **ссылку на портал доступа** для пакета Access. Дополнительные сведения см. [в разделе Совместное использование ссылки для запроса пакета Access](entitlement-management-access-package-settings.md). Если внешний пользователь просто посещает **myaccess.Microsoft.com** и не использует ссылку на портал «полный доступ», он увидит доступные пакеты доступа в своей организации, а не в вашей организации.

* Если внешний пользователь не может запросить доступ к пакету Access или не может получить доступ к ресурсам, обязательно проверьте [параметры для внешних пользователей](entitlement-management-external-users.md#settings-for-external-users).

* Если новый внешний пользователь, который ранее не входил в каталог, получит пакет для доступа, в который входит сайт SharePoint Online, его пакет будет отображаться как предоставленный неполностью, пока в SharePoint Online не будет подготовлена учетная запись. Дополнительные сведения о параметрах общего доступа см. [в статье Просмотр параметров внешнего общего доступа в SharePoint Online](entitlement-management-external-users.md#review-your-sharepoint-online-external-sharing-settings).

## <a name="requests"></a>Requests

* Когда пользователь хочет запросить доступ к пакету Access, убедитесь, что он использует **ссылку на портал доступа** для пакета Access. Дополнительные сведения см. [в разделе Совместное использование ссылки для запроса пакета Access](entitlement-management-access-package-settings.md).

* Если открыть портал "Мой доступ" в браузере с настроенным закрытым или анонимным режимом, может возникнуть конфликт при входе. Мы рекомендуем не использовать закрытый или анонимный режим для посещения портала "Мой доступ".

* Если пользователь, которого еще нет в каталоге, войдет на портал "Мой доступ" и запросит пакета для доступа, убедитесь, что он прошел проверку подлинности с использованием учетной записи организации. Учетная запись организации может быть в каталоге ресурсов или в каталоге, включенном в одну из политик пакета для доступа. Если учетная запись пользователя не принадлежит к организации или каталог проверки подлинности не включен в политику, пользователь не увидит пакет для доступа. Дополнительные сведения см. [в разделе запрос доступа к пакету Access](entitlement-management-request-access.md).

* Если для пользователя заблокирован вход в каталог ресурсов, пользователь не сможет запросить доступ на портале "Мой доступ". Чтобы пользователь мог запросить доступ, необходимо удалить блокировку входа из его профиля. Чтобы удалить блок входа, в портал Azure щелкните **Azure Active Directory**, выберите **Пользователи**, щелкните пользователя, а затем щелкните **профиль**. Измените раздел **Параметры** и измените параметр **блокировать вход** на **нет**. Дополнительные сведения см. в разделе [Добавление или обновление сведений о профиле пользователя с помощью Azure Active Directory](../fundamentals/active-directory-users-profile-azure-portal.md).  Можно также проверить, заблокирован ли пользователь из-за [политики защиты идентификации](../identity-protection/howto-identity-protection-remediate-unblock.md).

* Если пользователь одновременно является инициатором и утверждающим, на портале доступа он не увидит запрос пакета Access на странице **утверждения** . Это предусмотренное поведение — пользователь не может утвердить свой запрос. Убедитесь, что для запрашиваемого пакета для доступа согласно политике настроены дополнительные утверждающие. Дополнительные сведения см. в разделе [изменение параметров запросов и утверждений для пакета Access](entitlement-management-access-package-request-policy.md).

### <a name="view-a-requests-delivery-errors"></a>Просмотр ошибок доставки запроса

**Предварительная роль:** Глобальный администратор, администратор пользователей, владелец каталога, доступ к диспетчеру пакетов или диспетчер назначения пакетов Access

1. На портале Azure щелкните **Azure Active Directory** и выберите **Управление удостоверениями**.

1. В меню слева щелкните **доступ к пакетам** и откройте пакет Access.

1. Щелкните **запросы**.

1. Выберите запрос, который требуется просмотреть.

    Если запрос содержит ошибки доставки, состояние запроса будет **не доставлено или** **частично доставлено**.

    При возникновении ошибок доставки в области сведений запроса отобразится число ошибок доставки.

1. Щелкните число, чтобы просмотреть все ошибки доставки запроса.

### <a name="reprocess-a-request"></a>Повторная обработка запроса

Если после запуска запроса на повторную обработку пакета доступа возникла ошибка, необходимо подождать, пока система переобрабатывает запрос. Система несколько раз пытается выполнить повторную обработку в течение нескольких часов, поэтому в течение этого времени нельзя выполнить принудительную обработку. 

Можно выполнить только повторную обработку запроса, состояние **доставки которого не выполнено** или **частично доставлено** , а Дата завершения — менее одной недели. В противном случае кнопка **повторной обработки** будет неактивна.

![Кнопка повторной обработки неактивна](./media/entitlement-management-troubleshoot/cancel-reprocess-grayedout.png)

- Если ошибка исправлена в течение периода испытаний, состояние запроса изменится на **Доставка**. Запрос будет повторно обработан без дополнительных действий пользователя.

- Если ошибка не исправлена в течение окна пробных испытаний, то состояние запроса может быть **доставлено** **неудачно** или частично. Затем можно использовать кнопку **Повторная обработка** . Повторная обработка запроса будет продолжена через семь дней.

**Предварительная роль:** Глобальный администратор, администратор пользователей, владелец каталога, доступ к диспетчеру пакетов или диспетчер назначения пакетов Access

1. На портале Azure щелкните **Azure Active Directory** и выберите **Управление удостоверениями**.

1. В меню слева щелкните **доступ к пакетам** и откройте пакет Access.

1. Щелкните **запросы**.

1. Щелкните запрос, который требуется повторно обработать.

1. В области сведения о запросе щелкните **повторно обработать запрос**.

    ![Повторная обработка невыполненного запроса](./media/entitlement-management-troubleshoot/reprocess-request.png)

### <a name="cancel-a-pending-request"></a>Отмена ожидающего запроса

Можно отменять только ожидающий запрос, который еще не был доставлен или с которым произошел сбой доставки. В противном случае кнопка **Отмена** будет неактивна.

**Предварительная роль:** Глобальный администратор, администратор пользователей, владелец каталога, доступ к диспетчеру пакетов или диспетчер назначения пакетов Access

1. На портале Azure щелкните **Azure Active Directory** и выберите **Управление удостоверениями**.

1. В меню слева щелкните **доступ к пакетам** и откройте пакет Access.

1. Щелкните **запросы**.

1. Щелкните запрос, который нужно отменить.

1. В области сведения о запросе щелкните **отменить запрос**.

## <a name="multiple-policies"></a>Несколько политик

* Управление назначениями следует минимальным рекомендациям по обеспечению привилегий. Когда пользователь запрашивает доступ к пакету Access, имеющему несколько политик, которые применяются, управление назначением включает логику, позволяющую обеспечить более высокий приоритет или более конкретные политики на основе универсальных политик. Если политика является универсальной, управление назначением может не отображать политику для запрашивающей стороны или автоматически выбирать более ограниченную политику.

* Например, рассмотрим пакет Access с двумя политиками для внутренних сотрудников, в которых обе политики применяются к запрашивающему. Первая политика предназначена для конкретных пользователей, которые включают запрашивающий. Вторая политика предназначена для всех пользователей в каталоге, членом которых является запрашивающий. В этом сценарии первая политика автоматически выбирается для запрашивающего, так как он более четкий. Запрашивающему не предоставляется возможность выбора второй политики.

* Если применяются несколько политик, то политика, которая автоматически выбирается, или политики, отображаемые запрашивающему, основаны на следующей логике приоритета:

    | Приоритет политики | Область |
    | --- | --- |
    | P1 | Определенные пользователи и группы в каталоге или в конкретных подключенных организациях |
    | P2 | Все элементы в каталоге (за исключением гостей) |
    | P3 | Все пользователи в вашем каталоге (включая гостей) или определенные подключенные Организации |
    | P4 | Все настроенные подключенные организации или все пользователи (все подключенные Организации + все новые внешние пользователи) |
    
    Если любая политика имеет более высокий приоритет, категории с низким приоритетом не учитываются. Пример того, как в запрашивающей части отображается несколько политик с одинаковым приоритетом, см. в разделе [Выбор политики](entitlement-management-request-access.md#select-a-policy).
