# <a name="identity-protection-and-b2b-users"></a>Защита идентификации и пользователи B2B

Благодаря службе совместной работы Azure AD B2B организации могут применять политики на основе рисков для пользователей B2B, использующих защиту идентификации. Эти политики настраиваются двумя способами.

- Администраторы могут настраивать встроенные политики защиты идентификации на основе рисков, которые применяются ко всем приложениям, включая гостевых пользователей.
- Администраторы могут настраивать политики условного доступа, используя риск для входа в качестве условия, которое включает гостевых пользователей.

## <a name="how-is-risk-evaluated-for-b2b-collaboration-users"></a>Как оценивается риск для пользователей службы совместной работы B2B

Риск для пользователей службы совместной работы B2B оценивается в своем домашнем каталоге. Риск входа в реальном времени для этих пользователей оценивается в каталоге ресурсов при попытке доступа к ресурсу.

## <a name="limitations-of-identity-protection-for-b2b-collaboration-users"></a>Ограничения защиты идентификации для пользователей службы совместной работы B2B

Существуют ограничения в реализации защиты идентификации для пользователей службы совместной работы B2B в каталоге ресурсов, так как их удостоверения существуют в домашнем каталоге. Ниже приведены основные ограничения.

- Если гостевой пользователь запускает политику защиты идентификации пользователей для принудительного сброса пароля, они будут заблокированы. Этот блок из-за невозможности сбросить пароли в каталоге ресурсов.
- Гостевые пользователи не отображаются в отчете рискованные пользователи. Эта потеря видимости обусловлена оценкой рисков в домашнем каталоге пользователя B2B.
- Администраторы не могут отклонить или исправить пользователя службы совместной работы B2B в своем каталоге ресурсов. Эта утрата функциональности обусловлена тем, что администраторы в каталоге ресурсов не имеют доступа к домашнему каталогу пользователя B2B.

### <a name="why-cant-i-remediate-risky-b2b-collaboration-users-in-my-directory"></a>Почему я не могу исправить рискованные пользователи службы совместной работы B2B в моем каталоге?

Оценка рисков и их исправление для пользователей B2B происходят в домашнем каталоге. По этой причине гостевые пользователи не отображаются в отчете "рискованные пользователи" в каталоге ресурсов, а администраторы в каталоге ресурсов не могут принудительно установить безопасный сброс пароля для этих пользователей.

### <a name="what-do-i-do-if-a-b2b-collaboration-user-was-blocked-due-to-a-risk-based-policy-in-my-organization"></a>Что делать, если пользователь службы совместной работы B2B был заблокирован из-за политики на основе рисков в моей организации?

Если политика на основе риска блокирует рискованный пользователь B2B, ему нужно будет устранить этот риск в своем домашнем каталоге. Пользователи могут исправлять свои риски, выполняя безопасный сброс пароля в своем домашнем каталоге. Если для них не включен самостоятельный сброс пароля в домашнем каталоге, им необходимо обратиться к ИТ-специалистам Организации, чтобы администратор вручную отклонить риск или сбросить пароль.

### <a name="how-do-i-prevent-b2b-collaboration-users-from-being-impacted-by-risk-based-policies"></a>Разделы справки предотвратить влияние пользователей службы совместной работы B2B на политики, основанные на рисках?

Исключение пользователей B2B из политик условного доступа, основанных на рисках организации, помешает пользователям B2B затронуть или блокировать их оценку рисков. Чтобы исключить этих пользователей B2B, создайте группу в Azure AD, которая содержит всех гостевых пользователей вашей организации. Затем добавьте эту группу в качестве исключения для встроенных политик рисков для защиты идентификации и риска входа в систему, а также любых политик условного доступа, которые подвергаются риску входа пользователя в качестве условия.
