# <a name="tutorial-for-configuring-hypr-with-azure-active-directory-b2c"></a>Руководство по настройке ХИПР с помощью Azure Active Directory B2C

В этом примере учебника мы предоставляем рекомендации по настройке Azure AD B2C с помощью [хипр](https://get.hypr.com). Используя Azure AD B2C в качестве поставщика удостоверений, вы можете интегрировать ХИПР с любыми клиентскими приложениями, чтобы предоставить пользователям возможность проверки подлинности без пароля. ХИПР заменяет пароли шифрованием с открытым ключом, устраняя мошенничество, фишинг и повторное использование учетных данных.

## <a name="prerequisites"></a>Предварительные требования

Чтобы приступить к работе, вам потребуется:

- Подписка Azure AD. Если у вас нет подписки, вы можете получить [бесплатную учетную запись](https://azure.microsoft.com/free/).

- [Клиент Azure AD B2C](https://docs.microsoft.com/azure/active-directory-b2c/tutorial-create-tenant). Клиент связан с вашей подпиской Azure.

- ХИПР облачный клиент, получите бесплатную [пробную учетную запись](https://get.hypr.com/free-trial).

- Мобильное устройство пользователя, зарегистрированное с помощью API-интерфейсов ХИПР или ХИПР Device Manager в клиенте ХИПР. Например, для выполнения этой задачи можно использовать [пакет SDK для Хипр Java](https://docs.hypr.com/integratinghypr/docs/hypr-java-web-sdk) .

## <a name="scenario-description"></a>Описание сценария

Интеграция ХИРП включает следующие компоненты:

- Azure AD B2C — сервер авторизации, отвечающий за проверку учетных данных пользователя, также известный как поставщик удостоверений.

- Мобильные и веб-приложения, которые вы выбираете для защиты с помощью ХИПР и Azure AD B2C. ХИПР предоставляет надежный пакет SDK для мобильных устройств, а также мобильное приложение, которое можно использовать на платформах iOS и Android для проверки подлинности с паролем.

- Мобильное приложение ХИПР — мобильное приложение ХИПР можно использовать для выполнения этого примера, если предпочитается не использовать мобильные пакеты SDK в собственных мобильных приложениях.

- ХИПР API-интерфейсы RESTful. Вы можете использовать API ХИПР для регистрации и проверки подлинности пользователей и устройств. Эти интерфейсы API можно найти [здесь](https://apidocs.hypr.com).

Реализация показана на следующей схеме архитектуры.

![Снимок экрана для хипр — схема архитектуры](media/partner-hypr/hypr-architecture-diagram.png)

|Шаг | Описание |
|:-----| :-----------|
| 1. | Пользователь прибывает на странице входа. Пользователи выбирают вход и регистрацию и вводят на страницу имя пользователя.
| 2. | Приложение отправляет атрибуты пользователя в Azure AD B2C для обнаружения проверки.
| 3. | Azure AD B2C собирает атрибуты пользователя и отправляет атрибуты в ХИПР для проверки подлинности пользователя с помощью мобильного приложения ХИПР.
| 4. | ХИПР отправляет push-уведомление на зарегистрированное пользовательское мобильное устройство для быстрой идентификации с сертифицированной проверкой подлинности в Интернете (FIDO). Это может быть пользователь, биометрический или децентрализованный ПИН-код для печати пальца.  
| 5. | После того как пользователь подтвердит push-уведомление, пользователю предоставляется или запрещается доступ к приложению клиента на основе результатов проверки.

## <a name="configure-the-azure-ad-b2c-policy"></a>Настройка политики Azure AD B2C

1. Перейдите к [политике Azure AD B2C хипр](https://github.com/HYPR-Corp-Public/Azure-AD-B2C-HYPR-Sample/tree/master/policy) в папке Policy.

2. Чтобы скачать [LocalAccounts Starter Pack](https://github.com/Azure-Samples/active-directory-b2c-custom-policy-starterpack/tree/master/LocalAccounts) , следуйте этому [документу](https://docs.microsoft.com/azure/active-directory-b2c/custom-policy-get-started?tabs=applications#custom-policy-starter-pack)

3. Настройте политику для клиента Azure AD B2C.

>[!NOTE]
>Обновите указанные политики, чтобы они были связаны с конкретным клиентом.

## <a name="test-the-user-flow"></a>Тестирование потока пользователя

1. Откройте клиент Azure AD B2C и в разделе политики выберите **инфраструктура процедур идентификации**.

2. Выберите ранее созданный **сигнупсигнин**.

3. Выберите **запустить поток пользователя** и выберите параметры:

   А. **Приложение**: выберите зарегистрированное приложение (пример — JWT)

   Б. **URL-адрес ответа**: выберите **URL-адрес перенаправления**

   В. Выберите **Выполнить поток пользователя**.

4. Последовательно выберите процесс регистрации и создайте учетную запись.

5. ХИПР будет вызываться во время потока, после создания пользовательского атрибута. Если поток неполон, убедитесь, что пользователь не сохранен в каталоге.
