# <a name="azure-policy-assignment-structure"></a>Структура назначения в службе "Политика Azure"

Назначения политик используются политикой Azure для определения ресурсов, назначенных в ходе выполнения политик или инициатив. Назначение политики может определять значения параметров для этой группы ресурсов во время назначения, что позволяет повторно использовать определения политик, в которых используются те же свойства ресурсов, с разными требованиями к соответствию.

Для создания назначения политики используется JSON. Назначение политики содержит элементы для:

- display name
- Описание
- metadata
- режим принудительного применения
- определение политики
- Параметры

Например, в следующем JSON показано назначение политики в режиме _донотенфорце_ с динамическими параметрами:

```json
{
    "properties": {
        "displayName": "Enforce resource naming rules",
        "description": "Force resource names to begin with DeptA and end with -LC",
        "metadata": {
            "assignedBy": "Cloud Center of Excellence"
        },
        "enforcementMode": "DoNotEnforce",
        "policyDefinitionId": "/subscriptions/{mySubscriptionID}/providers/Microsoft.Authorization/policyDefinitions/ResourceNaming",
        "parameters": {
            "prefix": {
                "value": "DeptA"
            },
            "suffix": {
                "value": "-LC"
            }
        }
    }
}
```

Все примеры политик Azure приведены в [примерах политики Azure](../samples/index.md).

## <a name="display-name-and-description"></a>Отображаемое имя и описание

Используйте **DisplayName** и **Description** , чтобы определить назначение политики и предоставить контекст для его использования с определенным набором ресурсов. Максимальная длина **displayName** составляет _128_ символов, а **description** — _512_ символов.

## <a name="enforcement-mode"></a>Режим принудительного применения

Свойство **енфорцементмоде** предоставляет клиентам возможность проверить результат политики в существующих ресурсах без инициации воздействия политики или запуска записей в [журнале действий Azure](../../../azure-monitor/platform/activity-logs-overview.md). Этот сценарий обычно называется "What If" и соответствует практическим рекомендациям по развертыванию. **енфорцементмоде** отличается от [отключенного](./effects.md#disabled) действия, так как это предотвращает оценку ресурсов.

Это свойство имеет следующие значения:

|Режим |Значение JSON |введите |Исправить вручную |Запись журнала действий |ОПИСАНИЕ |
|-|-|-|-|-|-|
|Включено |значение по умолчанию |строка |Yes |Yes |Этот результат политики применяется во время создания или обновления ресурса. |
|Отключено |донотенфорце |строка |Yes |Нет | Этот результат политики не применяется во время создания или обновления ресурса. |

Если **енфорцементмоде** не указан в политике или в определении инициативы, используется значение _по умолчанию_ . [Задачи исправления](../how-to/remediate-resources.md) можно запустить для политик [deployIfNotExists](./effects.md#deployifnotexists) , даже если для **енфорцементмоде** задано значение _донотенфорце_.

## <a name="policy-definition-id"></a>Идентификатор определения политики

Это поле должно быть полным путем к имени политики или определению инициативы.
`policyDefinitionId` является строкой, а не массивом. Если несколько политик часто назначаются вместе, рекомендуется использовать [инициативу](./definition-structure.md#initiatives) .

## <a name="parameters"></a>parameters

Этот сегмент назначения политики предоставляет значения для параметров, определенных в [определении политики или в определении инициативы](./definition-structure.md#parameters).
Такая схема позволяет повторно использовать определение политики или инициативы с различными ресурсами, но проверять разные бизнес-значения или результаты.

```json
"parameters": {
    "prefix": {
        "value": "DeptA"
    },
    "suffix": {
        "value": "-LC"
    }
}
```

В этом примере параметры, ранее определенные в определении политики, `prefix` и `suffix`. Это конкретное назначение политики устанавливает `prefix` в **Отдел** а и `suffix` в **-LC**. Одно и то же определение политики можно многократно использовать с другим набором параметров для другого отдела, уменьшая дублирование и сложность определений политик, обеспечивая гибкость.
