# <a name="troubleshoot-the-windows-virtual-desktop-classic-management-tool"></a>Устранение неполадок в средстве управления виртуальными рабочими столами Windows (классическая модель)

>[!IMPORTANT]
>Это содержимое применимо к Виртуальному рабочему столу Windows (классическому), который не поддерживает объекты Azure Resource Manager для Виртуального рабочего стола Windows.

В этой статье описываются проблемы, которые могут возникнуть при развертывании средства управления виртуальными рабочими столами Windows, и способы их устранения.

## <a name="error-management-tool-services-configured-but-automated-setup-fails"></a>Ошибка: службы средства управления настроены, но автоматическая установка не выполняется

После успешной настройки служб для средства управления, но при сбое автоматической установки вы увидите следующее сообщение об ошибке:

```console
{"code":"DeploymentFailed","message":"At least one resource deployment operation failed. Please list deployment operations for details. Please see https://aka.ms/arm-debug for usage details.","details":[{"code":"Conflict","message":"{\r\n "status": "Failed",\r\n "error": {\r\n "code": "ResourceDeploymentFailure",\r\n "message": "The resource operation completed with terminal provisioning state 'Failed'."\r\n }\r\n}"}]}
```

Обычно это означает одно из следующих двух действий:

- Пользователь имеет разрешения владельца для своей подписки и глобального администратора на уровне клиента, но не может войти в Azure.
- Для параметров учетной записи пользователя включена многофакторная проверка подлинности.

Чтобы устранить эту проблему, выполните следующие действия:

1. Убедитесь, что пользователь, созданный для Azure Active Directory имени участника-пользователя, имеет уровень подписки "участник".
2. Войдите в <portal.azure.com> с учетной записью имени участника-пользователя, чтобы проверить параметры учетной записи и убедиться, что многофакторная проверка подлинности не включена. Если он включен, отключите его.
3. Перейдите на страницу разрешения виртуальных рабочих столов Windows и убедитесь, что сервер и клиентские приложения имеют согласие.
4. Ознакомьтесь с руководством по [развертыванию средства развертывания,](manage-resources-using-ui.md) Если проблема сохраняется, и повторно разверните средство.

## <a name="error-job-with-specified-id-already-exists"></a>Ошибка: задание с указанным ИДЕНТИФИКАТОРом уже существует

Если пользователь видит сообщение об ошибке "задание с указанным ИДЕНТИФИКАТОРом уже существует", это значит, что в параметре "имя приложения" при развертывании шаблона не было указано уникальное имя.

Чтобы устранить эту проблему, повторно разверните средство управления с заполненным параметром "имя приложения".

## <a name="delayed-consent-prompt-when-opening-management-tool"></a>Запрос отложенного согласия при открытии средства управления

При развертывании средства управления запрос согласия может не открыться сразу. Это означает, что для загрузки служба веб-приложений Azure занимает больше времени, чем обычно. Запрос должен появиться после загрузки Azure Web.

## <a name="the-user-cant-deploy-the-management-tool-in-the-east-us-region"></a>Пользователь не может развернуть средство управления в регионе "Восточная часть США"

Если клиент задает регион "Восточная часть США", он не может развернуть средство управления.

Чтобы устранить эту проблему, разверните средство управления в другом регионе. Повторное развертывание средства в другом регионе не должно повлиять на взаимодействие с пользователем.
