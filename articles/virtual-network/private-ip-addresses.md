# <a name="private-ip-addresses"></a>Частные IP-адреса
Частные IP-адреса разрешают обмен данными между ресурсами в Azure. 

Ресурсы могут быть:
* Службы Azure, такие как:
    * Сетевые интерфейсы виртуальных машин
    * Внутренние балансировщики нагрузки
    * Шлюзы приложений
* В [виртуальной сети](virtual-networks-overview.md).
* Локальная сеть через VPN-шлюз или канал ExpressRoute.

Частные IP-адреса разрешают обмен данными с этими ресурсами без использования общедоступного.

## <a name="allocation-method"></a>Способ выделения

Azure назначает ресурсам частные IP-адреса из диапазона адресов подсети виртуальной сети, в которой находится ресурс.

Azure резервирует первые четыре адреса в каждом диапазоне адресов подсети. Адреса не могут быть назначены ресурсам. Например, если диапазон адресов подсети — 10.0.0.0/16, адреса 10.0.0.0-10.0.0.3 и 10.0.255.255 недоступны. IP-адрес из диапазона адресов подсети можно назначить только одному ресурсу за раз. 

Существует два метода, в которых предоставляется частный IP-адрес:

- **Динамическое** — в Azure назначается следующий доступный неназначенный или незарезервированный IP-адрес в диапазоне адресов подсети. Например, новому ресурсу в Azure назначается адрес 10.0.0.10, если адреса в диапазоне 10.0.0.4–10.0.0.9 уже назначены другим ресурсам. 

    По умолчанию используется динамический способ выделения. После назначения динамические IP-адреса освобождаются, если используется сетевой интерфейс.
    
    * Deleted
    * Переназначено другой подсети в той же виртуальной сети.
    * Метод распределения изменяется на статический, и задается другой IP-адрес. 
    
    Если метод распределения изменяется с динамического на статический, в Azure по умолчанию назначается в качестве статического тот же адрес, который был до этого момента назначен динамически.

- **Статическое** — вы выбираете и назначаете любой неназначенный или незарезервированный IP-адрес в диапазоне адресов подсети. 

    Например, диапазон адресов подсети — 10.0.0.0/16 и адреса 10.0.0.4-10.0.0.9 назначаются другим ресурсам. Можно назначить любой адрес между 10.0.0.10-10.0.255.254. Статические адреса не освобождаются, пока сетевой интерфейс не будет удален. 
    
    При изменении метода распределения Azure назначает статический IP-адрес в качестве динамического IP-адреса. Переназначение выполняется, даже если адрес не является следующим доступным в подсети. Адрес изменяется, когда сетевой интерфейс назначается другой подсети.
    
    Чтобы назначить сетевой интерфейс другой подсети, измените метод выделения со статического на динамический. Назначьте сетевой интерфейс другой подсети, а затем снова измените метод выделения на статический. Назначьте IP-адрес из диапазона адресов новой подсети.
    
## <a name="virtual-machines"></a>Виртуальные машины

Один или несколько частных IP-адресов назначены одному или нескольким **сетевым интерфейсам**. Сетевые интерфейсы назначаются виртуальной машине [Windows](../virtual-machines/windows/overview.md?toc=%2fazure%2fvirtual-network%2ftoc.json) или [Linux](../virtual-machines/linux/overview.md?toc=%2fazure%2fvirtual-network%2ftoc.json) . Можно указать метод выделения для каждого частного IP-адреса: динамический или статический.

### <a name="internal-dns-hostname-resolution-for-virtual-machines"></a>Разрешение внутренних DNS-имен узла (для виртуальных машин)

По умолчанию виртуальные машины Azure настраиваются с помощью [DNS-серверов, управляемых Azure](virtual-networks-name-resolution-for-vms-and-role-instances.md#azure-provided-name-resolution) . Вы можете явно настроить пользовательские DNS-серверы. Эти DNS-серверы обеспечивают внутреннее разрешение имен для виртуальных машин, находящихся в одной виртуальной сети.

Сопоставление имени узла с частным IP-адресом виртуальной машины добавляется к DNS-серверам, управляемым Azure. 

Имя узла сопоставляется с основным IP-адресом основного сетевого интерфейса, если виртуальная машина имеет следующие преимущества.

* Несколько сетевых интерфейсов
* Несколько IP-адресов
* Оба варианта

Виртуальные машины, настроенные с помощью DNS под управлением Azure, разрешают имена узлов в той же виртуальной сети. Используйте пользовательский DNS-сервер для разрешения имен узлов виртуальных машин в подключенных виртуальных сетях.

## <a name="internal-load-balancers-ilb--application-gateways"></a>Внутренние балансировщики нагрузки и шлюзы приложений

Можно назначить частный IP-адрес конфигурации **интерфейса пользователя** :

* [Внутренняя подсистема балансировки нагрузки Azure](../load-balancer/load-balancer-internal-overview.md?toc=%2fazure%2fvirtual-network%2ftoc.json) (ilB)
* [Шлюз приложений Azure](../application-gateway/application-gateway-introduction.md?toc=%2fazure%2fvirtual-network%2ftoc.json) 

Этот частный IP-адрес выступает в качестве внутренней конечной точки. Внутренняя конечная точка доступна только для ресурсов в своей виртуальной сети и удаленных сетей, подключенных к ней. Можно назначить динамический или статический IP-адрес.

## <a name="limits"></a>Ограничения

Ограничения указаны по региону и по подписке. [Обратитесь в службу поддержки](https://portal.azure.com/#blade/Microsoft_Azure_Support/HelpAndSupportBlade) , чтобы увеличить ограничения по умолчанию до максимальных ограничений в зависимости от потребностей бизнеса.
