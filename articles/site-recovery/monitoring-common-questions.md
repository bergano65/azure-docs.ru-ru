# <a name="common-questions-about-site-recovery-monitoring"></a>Распространенные вопросы о мониторинге Site Recovery

В этой статье содержатся ответы на распространенные вопросы о мониторинге [Site Recovery](site-recovery-overview.md)Azure, использовании встроенного мониторинга Site Recovery и Azure Monitor (log Analytics).

## <a name="general"></a>Общее

### <a name="how-is-the-rpo-value-logged-different-from-the-latest-available-recovery-point"></a>Как регистрируется значение RPO, отличающееся от последней доступной точки восстановления?

Site Recovery использует многоэтапный асинхронный процесс для репликации компьютеров в Azure.

- На предпоследнем этапе репликации последние изменения на компьютере и метаданные копируются в учетную запись хранения журналов или кэша.
- Эти изменения вместе с тегом, который определяет восстанавливаемую точку, записываются в учетную запись хранения или управляемый диск в целевом регионе.
- Теперь Site Recovery может создать восстанавливаемую точку для компьютера.
- На этом этапе для всех изменений, отправленных в учетную запись хранения, было выполнено выполнение RPO. Иными словами, Целевая точка восстановления на этом этапе равна количеству времени, прошедшему с отметки времени, соответствующей восстанавливаемой точке.
- После этого служба Site Recovery получает данные, переданные из учетной записи хранения, и применяет их к дискам реплик, созданных для компьютера.
- Затем Site Recovery создает точку восстановления и делает ее доступной для восстановления при отработке отказа.
- Таким образом, последняя доступная точка восстановления Указывает метку времени, соответствующую последней точке восстановления, которая уже была обработана, и применена к дискам реплики.


Неправильное системное время на исходном компьютере репликации или на локальных серверах инфраструктуры приведет к смещению вычисленного значения RPO. Для точной отчетности RPO убедитесь, что системные часы на всех серверах и компьютерах показывают правильное время.



## <a name="inbuilt-site-recovery-logging"></a>Встроенное ведение журнала Site Recovery


### <a name="why-is-the-vm-count-in-the-vault-infrastructure-view-different-from-the-total-count-shown-in-replicated-items"></a>Почему число виртуальных машин в представлении инфраструктуры хранилища отличается от общего числа, показанного в разделе реплицированные элементы?

Представление инфраструктуры хранилища ограничивается сценариями репликации. Только компьютеры в выбранном в данный момент сценарии репликации включаются в счетчик для представления. Кроме того, считаются только виртуальные машины, настроенные для репликации в Azure. Виртуальные машины, для которых выполнена отработка отказа, или компьютеры, которые реплицируются обратно на локальный сайт, не учитываются в представлении.

### <a name="why-is-the-count-of-replicated-items-in-essentials-different-from-the-total-count-of-replicated-items-on-the-dashboard"></a>Почему количество реплицированных элементов в Essentials отличается от общего числа реплицированных элементов на панели мониторинга?

В число, показанное в Essentials, включаются только компьютеры, для которых выполнена начальная репликация. Общее число реплицированных элементов включает все компьютеры в хранилище, включая те, для которых выполняется начальная репликация.

### <a name="how-often-does-site-recovery-send-diagnostic-logs-to-azure-monitor-log"></a>Как часто Site Recovery отправляет журналы диагностики в журнал Azure Monitor? 

- Азуреситерековерирепликатионстатс и Азуреситерековерирековерипоинтс отправляются каждые 15 минут.  
- Азуреситерековерирепликатиондатауплоадрате и Азуреситерековерипротектеддискдатачурн отправляются каждые пять минут. 
- Азуреситерековерижобс отправляется по триггеру и завершению задания.
- Азуреситерековеревентс отправляется при каждом создании события. 
- Азуреситерековерирепликатедитемс отправляется при каждом изменении среды. Как правило, время обновления данных составляет 15 минут после изменения. 

### <a name="how-long-is-data-kept-in-azure-monitor-logs"></a>Как долго данные хранятся в журналах Azure Monitor? 

По умолчанию срок хранения составляет 31 день. Вы можете увеличить период в разделе **использование и оценка затрат** в рабочей области log Analytics. Щелкните **срок хранения данных**и выберите диапазон.

### <a name="whats-the-size-of-the-diagnostic-logs"></a>Каков размер журналов диагностики? 

Обычно размер журнала составляет 15-20 КБ.
