# <a name="azure-security-baseline-for-azure-cloud-services"></a>Базовые показатели безопасности Azure для облачных служб Azure

Этот базовый план безопасности применяет рекомендации из [теста безопасности Azure версии 1,0](../security/benchmarks/overview-v1.md) к облачные службы Microsoft Azure. Azure Security Benchmark содержит рекомендации по обеспечению безопасности облачных решений в Azure.
Содержимое группируются по **элементам управления безопасностью** , определенным в производительности системы безопасности Azure, и связанным рекомендациям, применимым к облачным службам. **Элементы управления** , неприменимые к облачным службам, были исключены.

 
Сведения о том, как облачные службы полностью сопоставлены с тестовым уровнем безопасности Azure, см. в [полном файле сопоставления базовых показателей безопасности облачных служб](https://github.com/MicrosoftDocs/SecurityBenchmarks/tree/master/Azure%20Offer%20Security%20Baselines).

### <a name="11-protect-azure-resources-within-virtual-networks"></a>1,1: защита ресурсов Azure в виртуальных сетях

**Руководство**. Создайте классическую виртуальную сеть Azure с отдельными общедоступными и частными подсетями, чтобы обеспечить изоляцию на основе доверенных портов и диапазонов IP-адресов. Эти виртуальные сети и подсети должны быть классическими ресурсами на основе виртуальной сети (классическая модель развертывания), а не текущими Azure Resource Manager ресурсами.  

Разрешает или запрещает трафик с помощью группы безопасности сети, которая содержит правила управления доступом на основе направления трафика, протокола, адреса источника и порта, а также адреса и порта назначения. Правила группы безопасности сети могут быть изменены в любое время, а изменения применяются ко всем связанным экземплярам.

Облачные службы Microsoft Azure (классическая модель) невозможно разместить в Azure Resource Manager виртуальных сетях. Однако виртуальные сети на основе диспетчер ресурсов и классические виртуальные сети на основе развертывания могут быть подключены через пиринг.

### <a name="12-monitor-and-log-the-configuration-and-traffic-of-virtual-networks-subnets-and-nics"></a>1,2. Мониторинг и запись конфигурации и трафика виртуальных сетей, подсетей и сетевых карт

**Руководство**. Документирование конфигурации облачных служб Azure и отслеживание изменений. Используйте файл конфигурации службы, чтобы указать количество экземпляров роли для развертывания для каждой роли в службе, значения параметров конфигурации и отпечатков для всех сертификатов, связанных с ролью. 

Если служба является частью виртуальной сети, необходимо указать сведения о конфигурации для сети в файле конфигурации службы, а также в файле конфигурации виртуальной сети. По умолчанию для файла конфигурации службы используется расширение .cscfg. Обратите внимание, что политика Azure не поддерживается для классических развертываний для принудительного применения конфигурации.

Задайте значения конфигурации облачной службы в файле конфигурации службы (cscfg) и определении в файле определения службы (CSDEF-файл). Используйте файл определения службы для определения модели службы для приложения. Определите роли, доступные для облачной службы, а также укажите конечные точки службы. Заносить в журнал конфигурацию для облачных служб Azure с помощью файла конфигурации службы. Любую перенастройку можно выполнить с помощью файла ServiceConfig. cscfg. 

Отслеживайте необязательное определение службы элемента NetworkTrafficRules, которое позволяет определить, какие роли могут обмениваться данными с указанными внутренними конечными точками. Настройте узел NetworkTrafficRules (необязательный элемент в файле определения службы), чтобы указать, как роли должны взаимодействовать друг с другом. Установите ограничения на то, какие роли могут обращаться к внутренним конечным точкам конкретной роли.  Обратите внимание, что определение службы не может быть изменено. 

Включите журналы потоков для групп безопасности сети и отправьте журналы в учетную запись хранения Azure для аудита. Отправьте журналы потоков в рабочую область Log Analytics и используйте Аналитика трафика, чтобы получить сведения о шаблонах трафика в клиенте Azure. Некоторые преимущества Аналитика трафика — возможность визуализировать сетевые активности, выявлять горячие участки и угрозы безопасности, анализировать шаблоны потоков трафика и указывать несетевые настройки.

### <a name="14-deny-communications-with-known-malicious-ip-addresses"></a>1.4. Запрет взаимодействия с известными опасными IP-адресами

**Руководство**. в Azure Cloud реализована многоуровневая Сетевая безопасность для защиты служб платформы от атак типа "отказ в обслуживании" (от атак DDoS). Защита Azure от атак DDoS является частью процесса непрерывного мониторинга облака Azure, который постоянно повышается за счет тестирования на проникновение. Эта защита от атак DDoS предназначена для того, чтобы выдерживать не только атаки извне, но и других клиентов Azure. 

Существует несколько различных способов блокировки или запрета обмена данными, помимо защиты на уровне платформы в облачных службах Azure. А именно: 

-  Создание задачи запуска для выборочного блокирования определенных IP-адресов
-  Ограничение доступа веб-роли Azure к набору указанных IP-адресов путем изменения файла web.config IIS

Запретите входящий трафик на URL-адрес по умолчанию или имя облачных служб, например *. cloudapp.NET. В качестве заголовка узла укажите пользовательское DNS-имя в разделе Конфигурация привязки сайта в файле определения облачных служб (* CSDEF-файл).

Настройка правила "запретить применение" для назначений администраторам классической подписки. По умолчанию после определения внутренней конечной точки данные могут передаваться из одной роли во внутреннюю конечную точку другой роли без каких-либо ограничений. Чтобы ограничить взаимодействие, необходимо добавить элемент NetworkTrafficRules к элементу ServiceDefinition в файле определения службы.

Центр безопасности использует машинное обучение для анализа приложений, выполняющихся на компьютерах, и создает список разрешений из этой аналитики. Эта возможность значительно упрощает процесс настройки и обслуживания политик списков разрешенных приложений, что позволяет выполнять следующие задачи:

- блокировать попытки запуска вредоносных приложений (в том числе тех, которые могли пропустить антивредоносные решения) и создавать оповещения об этих попытках;

- обеспечивать соответствие политике безопасности организации, которая предписывает использовать только лицензированное программное обеспечение;

- избегать использования нежелательного программного обеспечения в среде;

- избегать выполнения старых и неподдерживаемых приложений;

- избегать использования определенных программных средств, запрещенных в организации;

- управлять доступом к конфиденциальным данным через использование приложений для ИТ-отдела.
